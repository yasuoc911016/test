## 🧪 検証内容

Log Analytics（Microsoft Sentinel）の Entra **SignInLogs / AuditLogs テーブル**を用いて、以下を検証しました。

- エージェントのサインイン動作が記録されるか
- エージェントによる変更操作が監査できるか

---

## ① サインイン動作（SignInLogs）確認

### 結論

- Security Copilot のエージェントは、ユーザーとして対話的にサインインする設計ではありません
- そのため、Microsoft Entra の SignInLogs にはエージェントのサインインログは記録されません

**エージェントの実行**

実行日時：2/18 15:23

![](https://storage.googleapis.com/zenn-user-upload/292a744ad423-20260414.png)

**Advanced Huntingの実施**

```kql
SigninLogs
| where AppDisplayName contains "SecurityCopilot"
```

![](https://storage.googleapis.com/zenn-user-upload/575c671f97e1-20260414.png)

👉 サインインログの出力なし。これは設定不備や取得失敗ではなく、**仕様どおりの挙動**です。

---

### 補足：マネージド ID サインインログ

- 実行日時：2/18 15:23
- 結果：ログなし

👉 マネージド ID としてのログ出力もされないことを確認

![](https://storage.googleapis.com/zenn-user-upload/e01722a53013-20260414.png)

---

### サービスプリンシパルのサインインログ

エージェント ID：`f275aa57-a046-49a7-ba57-37d914c9bd4a`

![](https://storage.googleapis.com/zenn-user-upload/ea9e4470f428-20260414.png)

👉 ログ出力を確認。エージェントは **サービスプリンシパル（Agent ID）として動作** しており、サービスプリンシパルのサインインログとして記録されます。

---

### SignInLogs まとめ

| 確認項目 | 結果 |
|----------|------|
| ユーザーサインインログ | ❌ 記録なし |
| マネージド ID サインインログ | ❌ 記録なし |
| サービスプリンシパルサインインログ | ✅ 記録あり |

👉 ただし、サービスプリンシパルのサインインログには **「どのユーザーがエージェントを起動したか」は記録されません**。記録されるのはエージェント（アプリ）が API を呼び出した事実のみであるため、ユーザー単位の操作追跡には SignInLogs だけでは不十分です。ユーザー操作の監査には、後述の AuditLogs を併用してください。

---

## ② 変更操作（AuditLogs）について

### 結論

- エージェントによる変更操作は、Microsoft Entra の AuditLogs に記録される
- Microsoft Sentinel / Log Analytics から検知・監査可能であることを確認

---

### 検証 KQL

```kql
AuditLogs
| where LoggedByService == "Conditional Access"
| where tostring(InitiatedBy.app.displayName) startswith "SecurityCopilotAgentIdentity"
| project
    TimeGenerated,
    ActivityDisplayName,
    OperationName,
    InitiatedByApp = tostring(InitiatedBy.app.displayName),
    ServicePrincipalId = tostring(InitiatedBy.app.servicePrincipalId)
| order by TimeGenerated desc
```

### 検証結果

AuditLogs結果
![](https://storage.googleapis.com/zenn-user-upload/db24510f70f3-20260414.png)

### 確認できたログ

| 項目 | 値 |
|------|-----|
| ActivityDisplayName | Add conditional access policy |
| LoggedByService | Conditional Access |
| InitiatedBy | アプリ（SecurityCopilotAgentIdentity） |

👉 Security Copilot のエージェント（Agent ID）が実行主体として、Conditional Access ポリシーを作成・変更したことが監査ログ上で確認可能。

---

## 🎯 結論

| 観点 | 結果 | 詳細 |
|------|------|------|
| SignInLogs（ユーザー） | ❌ 記録なし | エージェントはユーザーとしてサインインしない設計（仕様通り） |
| AuditLogs（管理操作） | ✅ 記録あり | Conditional Access ポリシーの作成・変更は監査ログに記録される |
| 実行主体の特定 | ✅ 可能 | `InitiatedBy = app (SecurityCopilotAgentIdentity)` として確認可能 |
| Sentinel / Log Analytics での検知 | ✅ 可能 | エージェントによる管理操作は監査・検知可能 |

---

## 🧠 ベストプラクティス

Security Copilot エージェントの挙動監査においては：

- ⚠️ **SignInLogs（ユーザー）には記録されない** — サービスプリンシパルのサインインログでエージェントの動作自体は確認可能だが、起動したユーザーの特定はできない
- ✅ **AuditLogs を主軸に監査する** — 管理操作の内容と実行主体（SecurityCopilotAgentIdentity）が記録されるため、変更操作の監査はこちらを中心に据える

👉 SignInLogs と AuditLogs を組み合わせて監査するのが正しいアプローチです。
